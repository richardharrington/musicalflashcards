# Native Pitch Detection (Phase 4) — Implementation Spec

Status: agreed design, not yet implemented. Produced from a design interview on
2026-06-10. This document fleshes out Phase 4 of
[`pitch-detection-spec.md`](./pitch-detection-spec.md) — the port of the pitch
detection feature to the native (Expo / React Native) app. Everything decided
in that spec (§1 decisions, §2 verdict states, §6–§8 implementation addenda)
remains binding here; this document only adds what the native port needs. It
is written to be followed by an implementing agent without further design
input.

## 1. Decisions (settled — do not relitigate)

| Topic | Decision |
|---|---|
| Feature scope | **Both modes** (Practice and Tempo), full behavioral parity with web. |
| Platforms | iOS **and** Android. Android (the developer's physical phone) is the **primary acceptance platform** — full real-instrument criteria. iOS gets agent verification on the simulator plus a real-iPhone smoke pass on borrowed hardware. **Both are blocking** for Phase 4 completion. |
| Expo Go | Abandoned entirely. The app moves to **development builds** (`expo-dev-client`); `react-native-audio-api` cannot run in Expo Go. No conditional imports or degraded Expo Go mode. |
| Native projects | **CNG (Continuous Native Generation)**: `ios/` and `android/` are gitignored build artifacts, regenerated by `npx expo prebuild`. All native config lives in `app.json` + config plugins. Nothing under `ios/`/`android/` is ever hand-edited or committed. |
| Audio library | [`react-native-audio-api`](https://www.npmjs.com/package/react-native-audio-api) (Software Mansion), `^0.12.2` at time of writing — verify compatibility with Expo SDK 54 / RN 0.81 at implementation time and record the installed version in the addendum. |
| Capture mechanism | `AudioRecorder` **data callback** (`onAudioReady`) with a small `bufferLength`, accumulated in a **ring buffer** that emits overlapping `FRAME_SIZE` (2048) windows. The library's graph layer (`RecorderAdapterNode`, `AnalyserNode`) is deliberately **not** used — smallest API surface, no mic→speaker feedback risk. |
| Mic permission | Requested **explicitly in the native micSource** before starting the recorder, via an API that reports grant/denial uniformly on both platforms. Denial rejects `start()` with an error the shared pipeline hook maps to the Listen toggle's error state — mirroring web's `getUserMedia` flow exactly. |
| Code sharing | `usePitchPipeline` and the judge-wiring logic currently in web's `App.tsx` are **hoisted into the shared package**, parameterized by platform-specific pieces (micSource factory, error mapper). Web is refactored onto the shared hooks **first**, with zero behavior change, before any native code is written. |
| UI | Mirror web: same elements (mode selector, Listen toggle, live readout, practice prompt, mic hint), same placement relative to the staff, same colors and prominence rules (§7.5 of parent), expressed in the native app's existing component idiom (`Pressable` rows, `Switch`, etc.). No design exploration in this phase. |
| Verdict rendering | Already proven: VexFlow `setStyle` fill/stroke survives the SVG shim's serialization (verified by test on 2026-06-10). A permanent regression test locks this in (§3.1). |
| Staging | Three stages (§4): (a) shared refactor, web-verified; (b) dev-build infra + native micSource, readout proven on simulator/emulator; (c) both modes in native UI + device acceptance. |

## 2. Architecture

Everything between PCM frames and verdicts already exists in
`packages/shared/src/pitch/` and is consumed unchanged. The native work is one
new platform adapter plus UI, with the React glue promoted to shared:

```
[native] AudioRecorder (bufferLength ≈ 512) → ring buffer → overlapping 2048-sample windows
        │   micSource.ts — the only platform-specific audio code (mirrors web's contract)
        ▼
[shared] usePitchPipeline (hoisted from web; micSource factory injected)
        → PitchTracker → NoteEventTracker → PipelineFrame
        ▼
[shared] judge-session hook (hoisted from web App.tsx)
        → practice/tempo snapshots, judge lifecycle, bar-complete regeneration
        ▼
[native] RN components: Bar (SvgXml, decorated via buildFullMeasure), Listen
         toggle, live readout, mode selector, mic hint
```

### 2.1 Shared refactor (stage a)

**`usePitchPipeline` moves to `packages/shared/src/hooks/`.** Its platform
dependencies become parameters:

- a `createMicSource(onFrame)` factory with the existing contract —
  `start(): Promise<void>` / `stop(): void`, frames delivered as
  `(frame: Float32Array, sampleRate: number, atMs: number)`, processed
  synchronously and never retained, `stop()` safe to call repeatedly,
  in-flight `start()` guarded by a session counter;
- an error mapper `(err: unknown) => string` (web keeps its `DOMException`
  mapping; native maps permission/device errors). The user-facing strings stay
  identical across platforms: "Microphone access denied", "No microphone
  found", "Microphone unavailable".

**The judge wiring moves to a shared hook.** Web `App.tsx` currently owns
~130 lines of judge refs, snapshot state, lifecycle effects, and bar-complete
regeneration timing. All of it — including the subtle rules from the parent
spec (§7.3: fresh practice judge per bar/mode/listening change; §8.3: one
tempo judge spanning bars, `setTargets` on regeneration, `onBeat` on every
transition; `BAR_COMPLETE_DELAY_MS`, which moves to shared) — must live once,
in shared. The exact decomposition (one `useJudgedAppState` wrapping
`useAppState`, or a separate judging hook composed beside it) is the
implementer's call; the requirements are:

- both platform `App.tsx` files end up as pure UI over shared hooks;
- web's runtime behavior is bit-for-bit unchanged (verified per §4 stage a);
- the hooks work under both React 18 (web) and React 19 (native), like
  `useAppState` already does — no React-19-only APIs.

Follow the existing explicit-extension import style and export everything new
through `packages/shared/src/index.ts`.

### 2.2 Native micSource (`packages/native/audio/micSource.ts`)

- `createMicSource(onFrame)` with the contract above.
- `start()`:
  1. Request mic permission explicitly; on denial, reject with an error the
     native error mapper recognizes. (Implementer picks the API — e.g.
     `expo-audio`'s `requestRecordingPermissionsAsync` — on the criterion that
     it must report prior denial on **both** platforms without starting a
     recorder; do not rely on `AudioRecorder` itself to surface permission
     state, a known rough edge in the library.)
  2. Create `AudioRecorder` with `sampleRate: 48000` (use whatever rate the
     recorder actually reports when delivering buffers — the shared hook
     already recreates the tracker if `sampleRate` changes),
     `bufferLength: 512`, `channelCount: 1`.
  3. `onAudioReady`: append the buffer to a `FRAME_SIZE` (2048) ring buffer.
     Once the ring has filled, emit a full 2048-sample window every
     `EMIT_EVERY_N_BUFFERS` callbacks (default **2** → ~21 ms cadence at
     48 kHz, comparable to web's 60 Hz polling; a tunable knob if pitchy's CPU
     cost on the JS thread proves noticeable).
  4. `atMs` for an emitted window = the `when` timestamp of the latest
     contributing buffer plus that buffer's duration, in ms (i.e., the time of
     the window's last sample, relative to recording start — monotonic, which
     is all the trackers/judges require).
- `stop()`: stop/clean up the recorder, reset the ring buffer, bump the
  session counter.
- Reuse one `Float32Array` for emitted windows (same contract as web: the
  consumer processes synchronously).

### 2.3 Native build configuration

- Dependencies: `react-native-audio-api`, `expo-dev-client` (plus the
  permission API chosen in §2.2) in `packages/native`.
- `app.json`: add the `react-native-audio-api` config plugin with the iOS
  microphone usage description and Android `RECORD_AUDIO`; no background
  modes or foreground services (the mic only runs in the foreground).
- `.gitignore`: `packages/native/ios/`, `packages/native/android/`.
- Scripts: `npm run ios` / `npm run android` become `expo run:ios` /
  `expo run:android` (CNG prebuilds automatically). Update the **Commands**
  section of `CLAUDE.md` accordingly, noting that Expo Go no longer works.

### 2.4 Native UI

Mirror web's layout and semantics in the native app's existing component
style:

- **Mode selector** (`Practice | Tempo`) styled like the existing rest-count
  radio row; default **Tempo** (parent §7.3).
- **Listen toggle** near the staff: off / on / error states, error message
  inline; mic hint ("Check your microphone?", amber) beside it while tempo
  judging.
- **Live readout**: note name only (parent §7.5), prominent in practice mode,
  smaller in tempo mode; em-dash when silent.
- **Practice prompt** when practice mode is on but Listen is off.
- **`Bar`** gains the same decoration props as web's
  (`noteVerdicts`, `restWindowVerdicts`, `cursorIndex`), passed through to
  `buildFullMeasure`'s `MeasureDecorations`. No shim work is needed —
  `setStyle` colors serialize through `toSVGString` (verified; regression
  test in §3.1).
- Practice mode disables the beat clock and hides the metronome; tempo mode
  passes the beat-wrap regeneration delay while judging — both behaviors come
  free from the shared hooks.

## 3. Testing

### 3.1 Automated (vitest, shared package)

- **Shim verdict-styling regression test** (new, part of stage a): build a
  decorated measure (`correct`, `wrong`, cursor, `restViolated`), render it
  through the fake document exactly as native `Bar` does, and assert the
  verdict colors from `pitch/verdicts.ts` appear in `toSVGString` output.
  This locks in the one rendering assumption the native port depends on.
- Existing suites must keep passing; the hoisted hooks keep their logic in
  already-tested modules (judges/trackers are untouched).

### 3.2 Agent self-verification (before any human acceptance)

- **Stage a**: web re-verified with the §8.5 headless-Chrome oscillator
  harness from the parent spec (correct/green during the beat, restViolated,
  missed persisting through the wrap holdover, wrongOctave, mic hint,
  practice cursor walk). No real-instrument pass is owed here — the Phase 3
  pass was completed and confirmed by the developer on 2026-06-10.
- **Stages b/c**: dev build runs on the **iOS simulator** (uses the Mac's
  microphone; grant macOS mic permission) and the **Android emulator** (in
  extended controls → Microphone, enable "Virtual microphone uses host audio
  input"). Verify: permission grant and deny flows produce the right toggle
  states; the readout reacts to host-mic sound; in stage c, both modes judge
  plausibly end-to-end.

### 3.3 Human acceptance (blocking; defines Phase 4 done)

- **Android, physical device, real instrument — full criteria.** Practice
  mode: the parent spec's Phase 2 criteria (playing the displayed notes walks
  the cursor; new bar at the end; identical consecutive targets each need a
  re-strike; wrong octave amber, wrong note red, neither advances; denying
  permission shows the error state). Tempo mode: the parent spec's Phase 3
  criteria (at 60 BPM/defaults, the note played in its beat goes green,
  silence gray; a strike during a rest marks the rest red; talking/ambient
  noise does not; two silent bars raise the mic hint, cleared by the next
  qualifying strike).
- **iOS, borrowed iPhone, real instrument — smoke pass.** Install via a free
  provisioning profile (note: such installs expire after ~7 days — fine for a
  smoke pass). Verify: permission prompt and grant, readout tracks the
  instrument, one bar judges plausibly in each mode.

## 4. Build order & acceptance criteria

**Stage a — shared refactor (web only; no native changes).**
Hoist `usePitchPipeline` and the judge wiring into shared per §2.1; move
`BAR_COMPLETE_DELAY_MS` to shared; add the shim-styling regression test;
refactor web onto the shared hooks.
*Done when*: all vitest suites pass (including the new shim test); web
type-checks, builds, and passes the §3.2 headless-Chrome re-verification with
behavior identical to before the refactor; the native app still builds (it
doesn't consume the new hooks yet).

**Stage b — dev-build infra + native capture.**
Add dependencies, config plugin, CNG setup, and script changes per §2.3;
implement the native micSource per §2.2; wire the shared pipeline hook plus a
Listen toggle and live readout into the native app (no judging yet); update
`CLAUDE.md`'s commands.
*Done when*: `expo run:ios` and `expo run:android` produce working dev builds;
on both simulator and emulator, toggling Listen prompts for permission, the
readout tracks host-mic sound (including accidentals via
`midiToDisplayName`), denial shows the error state on the toggle, and
toggling off releases the mic.

**Stage c — modes, judging, and acceptance.**
Mode selector, practice cursor flow, tempo judging, verdict and rest-violation
coloring through `SvgXml`, mic hint, practice prompt, bar-complete
regeneration — all via the shared hooks, per §2.4.
*Done when*: the agent-level checks in §3.2 pass on simulator and emulator,
and the §3.3 human acceptance passes — full criteria on the Android device,
smoke pass on the borrowed iPhone. Both device passes are blocking.

## 5. Out of scope (decided against for this phase)

- Everything in the parent spec's §5 (intonation feedback, onset-timing
  grading, calibration, accidental targets, scoring).
- EAS / cloud builds, TestFlight/Play distribution, CI for native builds —
  local dev builds only.
- Background/lock-screen recording (no background modes are configured).
- Any UI redesign for phone ergonomics (e.g. music-stand-distance readout) —
  explicitly deferred until the port has proven itself on device.
- Keeping Expo Go working in any form.
- Consolidating the duplicated app constants (`NOTE_BOUNDARY_PAIRS`, etc.)
  between `web/src/main.tsx` and `native/App.tsx` — the keep-in-sync
  convention stands.

## 6. Known risks (accepted)

- `react-native-audio-api` is young and Android is where its rough edges
  live; the data-callback-only design (§1) minimizes the surface, and the
  explicit permission request avoids its known permission quirks. If
  `onAudioReady` proves unreliable on a platform, the fallback is the
  library's graph mode (recorder → `RecorderAdapterNode` → `AnalyserNode`
  polled on rAF, with a zero-gain node before the destination to avoid
  feedback) — a contained change inside `micSource.ts`.
- pitchy runs on the JS thread at ~21 ms cadence; if profiling shows it
  competing with React renders, raise `EMIT_EVERY_N_BUFFERS` (judging
  tolerates frame-granular timing per parent §6.4).
- Metro must transpile the shared package's raw TS and pitchy's ESM — both
  already work today (shared is already consumed by native; pitchy is a
  dependency of shared with no DOM usage).
- The first `expo run:*` per platform compiles the whole native project and
  is slow; subsequent builds are incremental.

## 7. Stage a implementation addendum (2026-06-10)

Stage a (shared refactor) is implemented per §2.1, with these judgment calls:

- **Decomposition: one `useJudgedAppState`** (`shared/src/hooks/`) wrapping
  `useAppState`, rather than a separate judging hook composed beside it. It
  owns mode state, the pipeline, both judge lifecycles, snapshot state, and
  bar-complete regeneration, and spreads `useAppState`'s return through its
  own. The verdict-selection ternaries (which snapshot feeds `Bar`, gated by
  the active judging mode) and the mic-hint condition were also hoisted into
  its return value (`noteVerdicts`, `restWindowVerdicts`, `cursorIndex`,
  `micHintVisible`), so the judge snapshots are not exposed at all — platform
  `App` components consume only presentation-ready values.
- **`usePitchPipeline` takes a params object**
  (`{ createMicSource, micErrorToMessage, onFrame }`); the mic-source contract
  types (`MicFrameCallback`, `MicSource`, `CreateMicSource`) live in that
  module and are exported from the package index. Both platform pieces are
  held in refs, so the hook tolerates (but does not expect) unstable
  identities.
- **Web's error mapper** lives in `web/src/audio/micSource.ts` as
  `micErrorToMessage` (it shares the mic source's error vocabulary); web's
  `hooks/usePitchPipeline.ts` is deleted. `BAR_COMPLETE_DELAY_MS` moved to
  `shared/src/pitch/constants.ts`.
- **Shim regression test** at
  `shared/src/rendering/__tests__/verdictStyling.test.ts`: renders through the
  fake document exactly as native `Bar` does and asserts all five verdict
  colors plus the cursor color survive `toSVGString`, that a violated rest
  colors its glyph, and that an undecorated measure contains none of the
  colors.
- **Verification**: all 56 vitest tests pass; web type-checks and builds. The
  §8.5 harness was rebuilt as a CDP script (oscillator-backed `getUserMedia`
  plus `Math.random = () => 0`, making every generated note G3/196 Hz —
  deterministic targets without reading them from the DOM). All 12 checks
  pass — Listen lifecycle, readout (G3 / em dash), correct-green during its
  beat, restViolated, missed-gray persisting through the wrap holdover,
  wrongOctave-amber, mic hint raise/clear, practice cursor walk over three
  identical targets with re-strikes, bar regeneration, practice prompt — and
  the identical harness passes 12/12 against the pre-refactor commit, pinning
  behavioral parity.
- **The harness is committed** as `packages/web/scripts/pitchHarness.mjs`,
  run via `npm run verify:pitch -w @musicalflashcards/web`. It is
  self-contained: no npm dependencies (raw CDP over Node's built-in
  `WebSocket`/`fetch`), builds the web package, serves `dist/` on a free port
  via vite preview, launches headless Chrome with an ephemeral profile, runs
  the 12 checks, and cleans up after itself. Env knobs: `CHROME_PATH`
  overrides the Chrome binary (default is the standard macOS install path);
  `APP_URL` skips build+serve and tests an already-running server. It is
  deliberately **not** wired into `npm test`: it takes ~40 s wall-clock and
  synchronizes to real 60 BPM beats, so it's an on-demand verification tool
  (re-run it whenever the shared hooks or web UI change), not a CI suite.
- The native app was confirmed to still bundle (`expo export`). Its
  `tsc --noEmit` has a pre-existing TS5097 noise floor (shared's
  explicit-extension imports vs. native's tsconfig, which lacks
  `allowImportingTsExtensions`) — present before this stage, unchanged by it,
  and irrelevant to Metro.
